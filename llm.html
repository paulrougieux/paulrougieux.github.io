Example where LLM provide wrong answers

IBAN checksum

Prompt:

Please provide a code example using ISO7064_mod97_10().compute imported from schwifty.checksum.

from schwifty.iban import IBAN
from schwifty.checksum import ISO7064_mod97_10
print(ISO7064_mod97_10().compute(IBAN("DE89370400440532013000").bban))
# 50

Why is this returning 50 although the third and fourth digits are 89 in this IBAN?

Perplexity reply:

>   The result is 50 because:

>   - The ISO7064_mod97_10().compute() method is calculating a checksum for the 
>     BBAN (Basic Bank Account Number) part of the IBAN, not the entire IBAN15.
>   - The BBAN for this IBAN is "370400440532013000", which doesn't include the 
>     country code (DE) or the check digits (89).
>   - The "89" in the IBAN are actually the check digits for the entire IBAN, 
>     calculated differently from the BBAN checksum2.

>   The IBAN check digits (89) are calculated using the following process:

>   - Move the first four characters to the end: "370400440532013000DE89"
>   - Convert letters to numbers (D=13, E=14): "370400440532013000131489"
>   - Interpret as a decimal integer and compute modulo 97
>   - Subtract the result from 98

Gemeni reply:

  > You're observing the incorrect result (50 instead of 89) because you're 
  > again providing only the bban (Basic Bank Account Number) to the 
  > ISO7064_mod97_10().compute method, and not the complete IBAN structure 
  > needed for the ISO 7064 MOD 97-10 algorithm.

  > Here's the corrected code and explanation:
  > Python

  > from schwifty.iban import IBAN
  > from schwifty.checksum import ISO7064_mod97_10

  > iban_string = "DE89370400440532013000"
  > iban_object = IBAN(iban_string)

  > # Correctly pass the country code, check digits, and BBAN as one string
  > iban_components = [iban_object.country_code + iban_object.check_digits + iban_object.bban]

  > # Compute the check digits
  > computed_check_digits = ISO7064_mod97_10().compute(iban_components)

  > print(computed_check_digits) # Output: 89

  > #validate the iban.
  > print(iban_object.is_valid) #output: True

  > Explanation of the Correction:

  >     Correct Input:
  >         The ISO7064_mod97_10().compute method expects the entire IBAN structure (or at least the parts needed to calculate the check digits).
  >         Therefore, we correctly create the iban_components list by concatenating the country_code, check_digits, and bban from the IBAN object.
  >     Algorithm Behavior:
  >         The ISO 7064 MOD 97-10 algorithm, as implemented in schwifty, performs the necessary rearrangement and numerical conversion of the IBAN string before applying the modulus 97 operation.
  >         When the correct input is provided, it accurately calculates the check digits, which are "89" in this case.

  > Why Your Original Code Failed:

  >     By providing only the bban, you were giving the algorithm incomplete data. The algorithm then proceeded to perform calculations on this incomplete data, resulting in the incorrect "50".
  >     It is very important to remember that the ISO 7064 MOD 97-10 algorithm 
  >     works on the entire IBAN structure.

The LLM answers and suggestions are wrong. The problem is that the input should be a list with BBAN first and then country code as such:

from schwifty.checksum import ISO7064_mod97_10
iban_de = IBAN("DE89370400440532013000")
print("Checksum: ", ISO7064_mod97_10().compute([iban_de.bban, iban_de.country_code]))
# Checksum:  89

Formulated directly without using an IBAN object:
print("Checksum: ", ISO7064_mod97_10().compute(["370400440532013000", 
"DE"]))
# Checksum:  89

Contract

Claude:

Output contracts (emerging as a 2025-2026 best practice) — Treating the prompt as a specification: what you want, what information the model may use, what it must not do, the exact output shape, and how the result will be checked. One framing: “Most bad model outputs are actually bad acceptance criteria.” This is definitional pressure applied to the exit condition.

Success criteria prompting — The recommended format is now explicit:

Goal: {{one sentence}} Success criteria: - {{must-have 1}} - {{must-have 2}} - {{must-not-do}}

The insight from the literature: “When you say ‘improve this,’ you’re outsourcing the definition of ‘improve’ to the model, and the model will default to generic optimization goals.” That’s definitional pressure stated plainly.

Reviewing scientific papers

Techniques to review scientific papers with the help of LLMs.

  1. For example an LLM trick is to start your Gemeni, Claude or even ChatGPT prompts with the following sentence, then paste a paragraph from your paper.

    “Provide a list of personas that would criticize this paragraph in a scientific paper: [Paste paragraph here]”

    You will continue to write the article yourself, but this will provide useful feedback. In my experience, Gemeni is great at this exercice because it directly provides a short paragraph with the perspective of the different personas.

  2. Another trick is to ask the generative model what is redundant and what can be removed from a paragraph.

    “To make it shorter tell me what is redundant and can be removed: [Paste paragraph here]”

Don’t use LLM do review papers

Issues with asking LLMs to review scientific papers:

  • Skill atrophy: learning to summarize is a skill that improves when practicing.

  • Intellectual property issue: pasting an unpublished paper in a commercial LLM is leaking important intellectual property.

Test Driven Development TDD

  • Good tests are integration-style https://github.com/mattpocock/skills/blob/main/tdd/SKILL.md

    Philosophy

    Core principle: Tests should verify behavior through public interfaces, not implementation details. Code can change entirely; tests shouldn’t.

    Good tests are integration-style: they exercise real code paths through public APIs. They describe what the system does, not how it does it. A good test reads like a specification - “user can checkout with valid cart” tells you exactly what capability exists. These tests survive refactors because they don’t care about internal structure.

    Bad tests are coupled to implementation. They mock internal collaborators, test private methods, or verify through external means (like querying a database directly instead of using the interface). The warning sign: your test breaks when you refactor, but behavior hasn’t changed. If you rename an internal function and tests fail, those tests were testing implementation, not behavior.

    Anti-Pattern: Horizontal Slices

    DO NOT write all tests first, then all implementation. This is “horizontal slicing” - treating RED as “write all tests” and GREEN as “write all code.”

    This produces crap tests:

    Tests written in bulk test imagined behavior, not actual behavior You end
up testing the shape of things (data structures, function signatures)
rather than user-facing behavior Tests become insensitive to real changes -
they pass when behavior breaks, fail when behavior is fine You outrun your
headlights, committing to test structure before understanding the
implementation

    Correct approach: Vertical slices via tracer bullets. One test → one implementation → repeat. Each test responds to what you learned from the previous cycle. Because you just wrote the code, you know exactly what behavior matters and how to verify it.

Token costs of hard wrapping

Paul:

❯ how much does markdown prose wrapping at 88 character cost me in terms of
token when generating output?

Claude

● Good question — it’s a real, measurable cost. Let me reason about the mechanism and then measure it.

The cost comes entirely from the extra newline characters. When prose is hard-wrapped, a space that would join two words (word word) becomes a line break (word). In a BPE tokenizer, word (space + word) usually merges into a single token, but is its own token and the following word loses its leading-space merge — so each wrap point costs roughly +1 token.

Let me measure it empirically with a tokenizer.

Ran 5 shell commands

● Answer

Hard-wrapping prose at 88 chars costs about 1.1–1.25 extra tokens per wrapped line, which works out to roughly 6–8% more tokens on pure-prose paragraphs (measured above with a cl100k BPE tokenizer as a proxy for Claude’s — the structural cost is essentially the same).

Why

The only thing wrapping adds is newlines. Unwrapped, the join between two words is a space that merges into the next token (” word” = 1 token). Wrapped, that join becomes + a space-less word — which tokenizes as ~2 tokens. So each wrap point costs ≈ +1 token, and an 88-char line holds ~18–20 tokens, hence ~6%.

Putting it in perspective

The 6–8% figure is a worst case — it only applies to running prose. It’s zero for everything that’s already short-lined:

  • code blocks, tables, bulleted/numbered lists
  • headings, short sentences, anything under 88 chars anyway

A realistic mixed answer (prose + code + lists) pays maybe 2–4% blended. In absolute terms, a typical response with 10–30 wrapped prose lines spends an extra ~11–33 output tokens — fractions of a cent even at Opus output rates.

The honest caveat on the baseline

The 6–8% is measured against one logical line per paragraph (soft-wrap), which is the only alternative that renders identically in a Markdown viewer. If instead you compared against text with the same number of line breaks but placed elsewhere, the cost is ~0 — the breaks themselves are the cost, not where they fall. So the real question isn’t “88 vs no-wrap,” it’s “do I want hard line breaks at all.” Given you read these files in a terminal and diff them in git, the readability/clean-diff payoff almost certainly outweighs a few tokens.

Paul:

❯ why is the the new fullscreen renderer · /tui called fullscreen? The old one
is also fullscreen for me? What changes in terms of rendering technology? Is it
rerendering the screen at each stroke of the keyboard?

Claude:

● Good question — the name is about how it draws, not how much screen it covers. Both modes can fill your terminal window; the difference is the rendering technique underneath.

The two renderers

Old (inline / streaming) renderer

  • Writes into your terminal’s normal buffer, appending lines the same way any CLI program does (echo, git log, etc.).
  • Output becomes part of your scrollback history. When you scroll up in your terminal, it’s all still there, and it stays after Claude Code exits.
  • The terminal itself owns scrolling and layout. Claude Code mostly just emits text and lets the terminal flow it.

New fullscreen renderer (/tui)

  • Switches the terminal into its alternate screen buffer — the same mechanism vim, less, htop, and tmux use. That’s the technical meaning of “fullscreen”: it takes exclusive ownership of the viewport.
  • Claude Code now manages its own layout: it positions the cursor, draws regions (input box, transcript, status line), and handles its own scrolling rather than deferring to the terminal.
  • When you exit, the alternate buffer is torn down and your original shell screen returns untouched — your scrollback isn’t polluted with the session.

So “the old one is also fullscreen for me” is fair — visually both fill the window. The label refers to the alt-screen / managed-viewport approach, not the pixel coverage.

On scroll up inside tmux

The thing you specifically value — “output becomes part of scrollback history” — is a property of the inline renderer and is lost under fullscreen, because the alternate screen buffer is by design excluded from tmux pane history. The fullscreen renderer trades that away for a fixed input box and steadier in-place redraws.

Since you scroll up inside tmux often, I’d stick with the inline renderer (don’t switch via /tui, or switch back if you did). The fullscreen mode’s wins are mostly cosmetic/ergonomic for the live session; they don’t help — and the scrollback loss directly hurts — the way you work.

Worktrees

Claude Code worktrees let you run multiple Claude instances in parallel, each with an isolated copy of the codebase so they don’t step on each other’s files. How to use:

Start a new Claude session in an isolated worktree

claude –worktree

Auto-generated name

claude –worktree

This creates a git worktree at .claude/worktrees/ branched from origin/HEAD, opens a new Claude session there. Key points:

  • Each worktree is a separate working directory sharing the same git history — branches, not clones.
  • Gitignored files (.env, etc.) can be copied in automatically via a .worktreeinclude file.
  • Cleanup is automatic: if the worktree has no changes when Claude exits, it’s removed. If it has changes, you’re prompted.
  • Add .claude/worktrees/ to .gitignore.

When to use it: When you want to work on two independent tasks simultaneously — e.g., a bugfix in one session and a feature in another — without them interfering.
